Overview

Это технический отчёт по проекту "video_converter" (Next.js + TypeScript) — небольшому фронтенд‑приложению, использующему FFmpeg (wasm) для конвертации видео в браузере. Цель отчёта — оценить текущее состояние реализации, качество кода, корректность интеграции FFmpeg.wasm, а также готовность к добавлению функций timelapse и обрезки (trimming). Отчёт содержит рекомендации и дорожную карту внедрения новых функций.

Architecture

- Фреймворк: Next.js (app router), React 19, TypeScript.
- State management: zustand (два стора: `useVideoDetailsStore` и `useVideoStore`).
- UI: компоненты из `@heroui/react` + Tailwind CSS конфигурация в проекте.
- FFmpeg: используется wasm‑сборка (`@ffmpeg/ffmpeg`, `@ffmpeg/core-mt`, `@ffmpeg/util`) — инициализация в хуке `useFFmpeg` и дальнейшая работа в компоненте `Header`.
- Структура файлов (ключевые модули):
    - `hooks/useFFmpeg.ts` — загрузка FFmpeg и предоставление рефа/логов.
    - `widgets/Header.tsx` — загрузка файла, запуск транскодинга, вывод лога и предпросмотр результата.
    - `utils/buildFFmpegArgs.ts` — собирает аргументы ffmpeg на основе состояния стора.
    - `utils/extractFrames.ts` + `features/TimeLines.tsx` — извлечение кадров и визуализация.
    - `features/CheckBoxes.tsx` — простые UI‑опции (flipHorizontal, flipVertical, removeMetadata).
    - `store/*.ts` и `interfaces/*` — модели состояния и типы.

Code Quality

Положительные стороны:

- Чёткая модульность: хуки, утилиты, компоненты и сторы разделены логично.
- Использование TypeScript и типов для стора и Frame (есть `interfaces` и `types`).
- Хорошая забота об освобождении URL (URL.revokeObjectURL) в большинстве мест.
- Простая, читабельная логика компонентов — минимальная бизнес‑логика в UI.

Замеченные проблемы и улучшения:

- Путаница в названиях состояния загрузки в `useFFmpeg`:
    - Переменная называется `isLoaded`, инициализируется как true, а после успешной загрузки ставится в false. Это вводит в заблуждение; поведение означает по сути "isLoading".
- В `buildFFmpegArgs` util жестко дергается `useVideoDetailsStore.getState()` — это делает функцию нечистой и усложняет её тестирование. Лучше принимать конфиг в аргументах.
- В `TimeLines` cleanup эффект использует замкнутую версию `frames` (может быть stale), и при изменении `file` может не корректно освобождать объекты. Сейчас автор отключил правило eslint и решил проблему частично, но лучше явно управлять ресурсами.
- Логирование (`logs` в `useFFmpeg`) аккумулируется бесконечно — может вырасти память логов при долгой работе.
- Почти нулевая обработка ошибок при загрузке FFmpeg (в `useFFmpeg.load`), а также именование/обработка состояния ошибок в UI/хук.
- Нет тестов и типичных проверок (unit/e2e).

FFmpeg Integration

Что реализовано:

- В `useFFmpeg` создаётся экземпляр FFmpeg (import { FFmpeg } from '@ffmpeg/ffmpeg'), подписываются логи и загружаются ресурсы core/wasm/worker из CDN (через `toBlobURL` и базовый `baseURL`).
- В `Header` используется `fetchFile` (из `@ffmpeg/util`) и методы на ffmpeg‑объекте: `writeFile`, `exec`, `readFile`.
- Результат записывается в Blob и создаётся URL для воспроизведения в <video>.

Возможные проблемы / замечания о корректности:

- API mismatch risk: в официальных примерах `@ffmpeg/ffmpeg` чаще используются функции `createFFmpeg({ log: true })`, `ffmpeg.load()`, `ffmpeg.FS('writeFile', ...)`, `ffmpeg.run(...)`, `ffmpeg.FS('readFile', ...)`. В коде проекта используются `new FFmpeg()` и методы `writeFile`/`exec`/`readFile`. Это может быть корректно для версии `@ffmpeg/ffmpeg`/`@ffmpeg/core-mt` — но нужно подтвердить, что используемые методы существуют в текущих версиях зависимостей (в `package.json` указаны версии `@ffmpeg/core-mt@^0.12.10`, `@ffmpeg/ffmpeg@^0.12.15`). Рекомендую явно проверить runtime/типизацию или заменить реализацию на официально документированный API (createFFmpeg + FS + run).
- Загрузка движка через `toBlobURL` и CDN — рабочая стратегия, позволяет использовать multi‑thread core (`core-mt`). Отмечу, что `baseURL` захардкожен на конкретную версию (`0.12.6`) внутри `useFFmpeg`, а `package.json` использует более новые версии — это потенциальный источник несоответствий. Стоит унифицировать версии или загружать из того же пакета, который установлен.
- Worker/wasm загружаются корректно в браузер, но нет механизма контроля использования памяти/освобождения ядра при размонтировании.
- Прогресс конвертации не отображается (хотя логи подписаны — логи выводятся в UI, но нет парсинга прогресса для прогресс-бара).

Current Feature Status

Реализовано (полностью/рабоче):

- Загрузка FFmpeg (веб‑сборка) и инициализация (в большинстве случаев).
- Загрузка локального файла пользователем и отображение предварительного просмотра в <video> (используется URL.createObjectURL).
- Транскодинг в браузере: запись входного файла во внутреннюю FS, выполнение команды, чтение `output.mp4` и отображение результата.
- Простые трансформации: flipHorizontal / flipVertical (реализована сборка фильтров в `buildFFmpegArgs` и UI чекбоксы).
- Удаление метаданных (через `-map`, `-map_metadata -1`, `-map_chapters -1`).
- Извлечение кадров для таймлайна (`utils/extractFrames.ts` + `features/TimeLines.tsx`) — извлекает N кадров и показывает миниатюры.

Реализовано частично / требует доработки:

- Параметры кодека и качества (preset, crf) есть в сторе, но UI для управления ими отсутствует.
- Логирование: логи собираются и показываются как plain text, но управление/фильтрация/прогресс отсутствуют.
- Управление ресурсами FFmpeg: нет явного teardown/terminate/проверки ошибок загрузки ядра.

Отсутствует / не реализовано:

- Timelapse (ускорение видео) — нет параметра в UI/сторе и нет сборки аргументов для установки setpts/atempo и т.д.
- Trimming (обрезка) — нет UI/хранения времени начала/конца и соответствующей генерации аргументов (-ss/-to/-t).
- Поддержка больших файлов: нет разбивки, показ прогресса, предупреждений по памяти.
- Тесты, CI, обработка ошибок и пользовательские сообщения при сбоях.

Missing or Weak Parts

1. Версионирование и API‑согласованность
    - В `useFFmpeg` используется жёсткий `baseURL` с версией 0.12.6, тогда как в package.json указаны другие версии пакетов. Это может привести к несоответствию между API локальной библиотеки и загружаемого core.
2. Нейминг состояния загрузки
    - `isLoaded` invert. Лучше: `isLoading` (boolean, true while loading) или `isReady` (false while loading, true when ready).
3. Управление ресурсами / утечки
    - `logs` растут без ограничений.
    - Неочевидный lifecycle для экземпляра FFmpeg: нет teardown/terminate.
    - `TimeLines` может не корректно освобождать frame URLs при быстрых сменах файлов из‑за stale closure.
4. UX и контроль выполнения
    - Нет прогресс‑бара транскодинга (нельзя оценить длительность операции).
    - Отсутствие UI для настройки `preset`, `crf`, а также новых параметров (timelapse speed, trim start/end).
5. Тестируемость и чистота кода
    - `buildFFmpegArgs` обращается к сторам напрямую. Лучше сделать чистую функцию, принимающую конфиг.
    - Нет модульных тестов.

Readiness for Timelapse

Текущая база позволяет внедрить timelapse с относительно небольшими изменениями:

- Что уже есть, полезное для timelapse:
    - Механизм записи файла и вызова ffmpeg с произвольными args (`Header.transcode`).
    - Утилита `buildFFmpegArgs` для формирования аргументов.
    - UI/стор для опций (есть `useVideoDetailsStore`) — в сторе можно добавить параметры для timelapse.
- Что нужно добавить:
    1. Новые параметры в сторе: `timelapseFactor` (number, >0; например 2,4,8) и/или `timelapseMode` (video/audio handling).
    2. Правки в `buildFFmpegArgs` либо расширение его интерфейса: если timelapseFactor > 1, добавить фильтр `setpts=PTS/<factor>` для видео. Для аудио — использовать цепочку фильтров `atempo` (замечание: atempo поддерживает 0.5–2.0 за один фильтр; для большего ускорения нужно цепочку множителей, либо отрезать/удалить аудио).
    3. UI‑элементы для выбора множителя скорости (селектор/ползунок) и опционально предварительный просмотр кадров/временной шкалы.
- Риски и рекомендации:
    - Аудио: если factor высокий (>2), `atempo` нужно применить несколько раз (например для 4x → atempo=2,atempo=2). Это нужно генерировать автоматически.
    - Потеря качества/проблемы с синхронизацией: при сильном ускорении могут быть проблемы со звуком; при желании можно просто удалить аудио или оставить pitch‑correct via more advanced processing (в рамке фронтенда это сложно).
- Оценка готовности: проект структурно готов к внедрению timelapse. Оценка: 70–80% готовности — требуется добавление пары параметров в стор, расширение builder'а аргументов и UI.

Readiness for Video Trimming

Trimming (обрезка по времени) также можно добавить легко:

- Что нужно добавить:
    1. Параметры в сторе: `trimStart` (секунды или timecode), `trimEnd` / `duration`.
    2. UI: контролы для выбора диапазона (например два input'а, или визуальный таймлайн с draggable handles — `TimeLines` может послужить основой).
    3. В `buildFFmpegArgs` вставить `-ss` и `-to` или `-t` в правильных местах. Рекомендации:
        - Для быстрого seek (иногда менее точного) — ставить `-ss` перед `-i`.
        - Для точного (frame‑accurate) — ставить `-ss` после `-i` и использовать `-copyts`/прочие опции, но это дороже по времени.
    4. Обработать случаи: только start, только duration, start+end.
- Риски:
    - Порядок аргументов влияет на производительность/точность. Нужно документировать выбранную стратегию.
- Оценка готовности: 80–90% — большинство механизмов уже на месте, требуется UI + аргументы.

Recommendations

1. Привести API загрузки FFmpeg в соответствие с версиями:
    - Синхронизовать версии `baseURL` в `useFFmpeg` с `package.json` либо загрузить core из пакета (если возможно).
    - Проверить, существуют ли методы `writeFile/exec/readFile` у используемого FFmpeg‑объекта; при несоответствии переписать на официальный паттерн: `createFFmpeg`, `ffmpeg.load()`, `ffmpeg.FS('writeFile', ...)`, `ffmpeg.run(...)`, `ffmpeg.FS('readFile', ...)`.

2. Убрать побочные эффекты из утилит:
    - Изменить `buildFFmpegArgs` на чистую функцию: принимать конфиг (опции flips, preset, crf, timelapseFactor, trimStart/End и т.д.) вместо глобального `useVideoDetailsStore.getState()` внутри.

3. Улучшить управление состоянием загрузки/ошибок:
    - Переименовать `isLoaded` → `isLoading`/`isReady` и добавить `error` в `useFFmpeg`.
    - Добавить явный teardown: при размонтировании попытаться завершить worker/освободить ресурсы (если API позволяет).

4. Прогресс и UX:
    - Парсить сообщения лога для получения прогресса (ffmpeg выводит frame/time/bitrate в лог) и отображать процент выполнения/ProgressBar.
    - Ограничить количество строк логов (например последние 200 записей) или реализовать cycled buffer.

5. Memory / large files
    - Добавить предупреждения о больших файлах, замер использования памяти.
    - Рассмотреть опцию server‑side транскодинга для больших видео (или Chunked processing), если браузерный режим не подходит.

6. TimeLines / Frame extraction
    - В `TimeLines` вынести логику извлечения в более контролируемый хук, корректно освобождать кадры при сменах и тестировать race–conditions.

7. Tests & CI
    - Добавить unit‑tests для `buildFFmpegArgs` и e2e тесты для ключевых пользовательских сценариев.

Implementation Roadmap (пошагово)

Минимальный план внедрения timelapse + trimming с приоритетами.

Шаг 0 — короткая подготовка (1–2 дня)

- Проверить версии пакетов и API реального FFmpeg‑объекта в рантайме.
- Исправить `isLoaded` на `isLoading`/`isReady` (малое изменение, повышает ясность).

Шаг 1 — рефактор: чистая генерация аргументов (1 день)

- Изменить `buildFFmpegArgs` так, чтобы она принимала аргумент типа:
  { input: string, output: string, flipHorizontal?: boolean, flipVertical?: boolean, preset?: string, crf?: number, removeMetadata?: boolean, timelapseFactor?: number, trimStart?: number, trimEnd?: number }
- Сделать unit‑тесты для этой функции.

Шаг 2 — стор и UI (1–2 дня)

- Расширить `useVideoDetailsStore`/`interfaces` новыми полями: `timelapseFactor`, `trimStart`, `trimEnd`.
- Добавить UI-компоненты:
    - Контрол для speed (timelapse): селектор 1x/1.5x/2x/4x/8x (или ввод числа).
    - Контрол для trimming: два input'а для начала и конца (в секундах) и/или визуальный range на `TimeLines` с draggable handles.

Шаг 3 — реализовать аргументы ffmpeg (0.5–1 день)

- Timelapse: добавить в args фильтр `-vf setpts=PTS/<factor>`.
    - Для audio: если factor === 1 → ничего; else if 0 < factor <= 2 → `-filter:a atempo=<factor>`; else цепочка atempo (пример: for factor=4 → atempo=2,atempo=2). Можно также опционально drop audio (`-an`).
- Trimming:
    - Для быстрого подхода: если trimStart задан и trimEnd задан → prepend `-ss <start>` BEFORE `-i`, and append `-to <end>` AFTER? Рекомендую для простоты использовать `-ss` перед `-i` and `-t` after `-i` as `-t = trimEnd - trimStart`.
    - Document behaviour (fast vs accurate trimming) in UI.

Шаг 4 — UX: прогресс и улучшенный лог (1–2 дня)

- Парсить ffmpeg логи и отображать прогресс.
- Ограничить буфер логов и показать ошибки пользователю.

Шаг 5 — тестирование и оптимизация (1–3 дня)

- Тесты unit/e2e, проверка на нескольких видео форматах.
- Проверить память и поведение на больших видео (30–60 мин клипы).

Примеры ffmpeg аргументов

- Timelapse (video ускорить в 4 раза, ускорить audio в 4x → chain atempo):
    - Видео: -vf "setpts=PTS/4"
    - Аудио: -filter:a "atempo=2,atempo=2" (4 = 2\*2)
    - Итог: ['-i', 'in.mp4', '-vf', 'setpts=PTS/4', '-filter:a', 'atempo=2,atempo=2', 'output.mp4']

- Trimming (быстрая команда с -ss перед -i и -t после):
    - Для обрезки 10s..30s (duration 20s): ['-ss', '10', '-i', 'in.mp4', '-t', '20', '-c', 'copy', 'output.mp4']
    - Для точного рампинга (если нужна точность) используйте `-ss` после `-i` и не указывайте `-c copy`.
